# ER Export Framework

Фреймворк на базе Airflow для автоматизированной выгрузки данных из **ClickHouse** в **S3** с последующей нотификацией системы **TFS** через **Kafka**.

## Обзор архитектуры

Система построена на принципах **Metadata-driven development**. Для добавления новой таблицы в поток выгрузки не требуется написание кода — достаточно добавить запись в управляющую таблицу в ClickHouse.

### Основной поток данных:
1. **ClickHouse** — источник данных.
2. **S3 (Ceph)** — промежуточное хранилище для CSV и ZIP архивов.
3. **Kafka** — канал уведомлений для TFS (отправка XML-сообщений).
4. **TFS** — целевая система потребления данных.

## Режимы работы (Environment Modes)

| Режим | Описание | Признак |
| :--- | :--- | :--- |
| **SIGMA** | "Боевой" режим (Production) | `ER_MODE=SIGMA` (по умолчанию) |
| **ALPHA** | Тестовый режим (Dev/Test) | `ER_MODE=ALPHA` или наличие `AIRFLOW__CTL_PIN` |

В режиме **ALPHA** автоматически:
- Используются секреты из Vault (`/vault/secrets/application`).
- Применяется лимит `LIMIT 100` на выгрузку (для сред `uat`, `qa`, `ift`, `dev`).
- Пропускается отправка уведомлений в Kafka.

## Структура файлов

| Файл | Описание |
| :--- | :--- |
| `er_export.py` | **Фабрика DAG-ов**. Динамически создаёт DAG для каждой активной выгрузки. Содержит всю бизнес-логику: SQL-билдеры, хелперы и задачи выгрузки. |
| `er_sync.py` | **DAG синхронизации**. Синхронизирует `export.er_wf_meta` с Airflow Variable (`datalab_er_wfs`). Также отвечает за создание пула `datalab_export_er`. |
| `er_config.py` | **Конфигурация и Утилиты**. Константы, маппинг типов и локальные хелперы (`obj_load`, `obj_save`, `add_note`), изолирующие фреймворк от внешних зависимостей. |
| `er_wf_meta.sql` | **DDL**. Структура управляющей таблицы `export.er_wf_meta` и примеры заполнения. |

## Управление метаданными (Persistence)

Метаданные выгрузок хранятся в Airflow Variables. Для работы используются функции:
- **`obj_load(key)`**: Загрузка JSON-объекта из переменной.
- **`obj_save(key, data)`**: Сохранение с проверкой изменений (запись только при отличии данных). Автоматически обновляет `Description` переменной, добавляя метаданные:
  - `ts`: Время последнего изменения.
  - `len`: Количество элементов (таблиц).
  - `size`: Человекочитаемый размер JSON-строки.

## Как добавить новую выгрузку

Все настройки хранятся в таблице `export.er_wf_meta`.

### Автоматическое определение полей:
Если в поле `fields` указано `['*']` или пусто, система автоматически выполнит `DESCRIBE TABLE` и включит все доступные колонки ClickHouse в `.meta` файл и CSV-выгрузку.

### Первая выгрузка (Bootstrap):
Для новых таблиц не нужно вручную создавать запись в `extract_current`. Система автоматически инициализирует состояние, используя `lower_bound` из реестра при первом запуске DAG.

### Пример добавления:

```sql
INSERT INTO export.er_wf_meta
    (extract_name, db_name, replica, schema_name, strategy, uk, fields, sql_from, sql_where, increment, selfrun_timeout, auto_confirm)
VALUES (
    'my_table_name',           -- Имя выгрузки
    'my_database',             -- БД в ClickHouse
    'hrplatform_datalab',      -- Реплика (маршрут в TFS)
    'target_schema',           -- Схема в целевой системе
    'FULL_UK',                 -- Стратегия (FULL_UK/DELTA_UK/...)
    ['id'],                    -- Unique Key
    ['*'],                     -- Авто-обнаружение всех полей
    'my_database.my_table',    -- Источник (FROM)
    '{condition}',             -- WHERE-условие
    60,                        -- Инкремент дельты (сек)
    10,                        -- Таймаут автозапуска (мин)
    1                          -- 1 = автоподтверждение, 0 = ждать Kafka
);
```

## Процесс выгрузки (Pipeline)

DAG-и имеют плоскую структуру задач (без TaskGroups) для стабильного отображения в UI Airflow:

1. **init** — Инициализация состояния. Поддерживает автоматический bootstrap для новых таблиц.
2. **build_meta** — Генерация `.meta` файла. Поддерживает авто-обнаружение колонок через `DESCRIBE`.
3. **export_to_s3** — Нативная выгрузка ClickHouse → S3 (`HrpClickNativeToS3ListOperator`).
4. **pack_zip** — **Потоковая упаковка**: стримит данные из S3 в S3, упаковывая в ZIP "на лету" (без загрузки всего файла в память).
5. **notify_tfs** — Нотификация TFS через Kafka. Пропускается в режиме **ALPHA**.
6. **wait_confirm** — Ожидание подтверждения (AwaitMessageSensor) или заглушка (EmptyOperator).
7. **save_status** — Фиксация истории в `export.extract_history`.
8. **schedule_next** — Автоматический перезапуск (Self-run) до актуализации данных.

## Технические детали

- **Pool**: Задачи выполняются в пуле `datalab_export_er` (20 слотов).
- **Стриминг**: ZIP-архивация потребляет минимум памяти благодаря библиотеке `stream_zip` и multipart-загрузке в S3.
- **Изоляция**: Фреймворк минимизирует зависимости от общих `plugins`, используя собственные хелперы в `er_config.py`.
